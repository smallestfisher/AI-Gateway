## 2. 请求生命周期（时序）

以「Claude Code 经网关访问一个 OpenAI Chat 兼容上游」为例（最难的一种：协议跨族 + 工具 + 推理）：

```
Client                Ingress            Router            Egress             Upstream
  │   POST /v1/messages │                   │                 │                    │
  │  (messages wire)    │                   │                 │                    │
  │────────────────────▶│                   │                 │                    │
  │                     │ request_id 生成    │                 │                    │
  │                     │ AuthN(API_KEY→user)│                 │                    │
  │                     │ 限流/配额检查(Redis)│                 │                    │
  │                     │ Adapter.DecodeRequest              │                    │
  │                     │ messages wire ──▶ UnifiedRequest   │                    │
  │                     │──────────────────▶│                 │                    │
  │                     │                   │ Resolve(alias)  │                    │
  │                     │                   │  →[Ch_A,Ch_B..] │                    │
  │                     │                   │ Circuit 过滤    │                    │
  │                     │                   │ weighted 选 Ch_A│                    │
  │                     │                   │────────────────▶│                    │
  │                     │                   │                 │ BuildUpstream body │
  │                     │                   │                 │ (IR→chat wire)     │
  │                     │                   │                 │ Profile 注入Header │
  │                     │                   │                 │ 模型名重写         │
  │                     │                   │                 │───────────────────▶│
  │                     │                   │                 │                    │ 处理
  │                     │                   │                 │◀─────SSE stream────│
  │                     │                   │                 │ 逐 chunk:          │
  │                     │                   │                 │ DecodeUpstreamStream│
  │                     │                   │                 │  chat delta ──▶ IR │
  │                     │                   │                 │ EncodeStreamEvent  │
  │                     │                   │                 │  IR ──▶ messages ev│
  │◀─────────── SSE event stream (messages 格式) ────────────│                    │
  │                     │                   │                 │ 指标采样→Redis     │
  │                     │                   │                 │ 日志→异步队列      │
  │                     │                   │                 │ 配额扣减           │
```

**重试 / Failover（在 Egress 内闭环）**：

```
尝试 Ch_A ──失败(可重试: 5xx/超时/熔断)──▶ 回退 Ch_B ──失败──▶ Ch_C ...
   │ 不可重试(4xx 业务错误/内容安全)──▶ 直接把错误按客户端协议返回，不 failover
   │ 流式已发首块后失败 ──▶ 不再 failover（已对客户端产生输出），转错误事件
```

> **关键约束**：failover 只允许在**尚未向客户端发送任何响应字节**时进行。一旦首块/首字节已发出，后续失败只能以「错误事件」收尾，不能静默重试——否则客户端会收到拼接错乱的流。

## 3. 分层与数据流总览

```
        ┌─────────────────── 协议面（多方言） ───────────────────┐
        │  openai_chat  openai_responses  anthropic_messages  ... │
        └───────────────┬─────────────────────────────────────── ┘
   DecodeRequest / EncodeResponse        ◀ 入口 Adapter（client 协议）
                        │
                        ▼
        ┌──────────────────────────────────────────────────────┐
        │                 Unified IR (block 模型)              │
        │   UnifiedRequest ⇄ UnifiedResponse ⇄ StreamEvent     │
        └───────────────┬───────────────────────────────────────┘
  BuildUpstream / DecodeUpstream         ◀ 出口 Adapter（provider 协议）
                        │
                        ▼
        ┌─────────────────── 协议面（多方言） ───────────────────┐
        │  openai_chat  anthropic_messages  google_gemini  ...   │
        └────────────────────────────────────────────────────────┘
```

核心思想：**入口协议**（客户端用的）与**出口协议**（上游用的）相互独立。任意入口 × 任意出口的组合都由 IR 居中翻译，组合数 = M×N 通过 IR 降为 M+N。

## 4. 部署拓扑（生产级）

```
                        ┌──────────────┐
                        │   LB / WAF   │  ── TLS 终结、限流、防滥用
                        └──────┬───────┘
               ┌───────────────┼───────────────┐
               ▼               ▼               ▼
        ┌──────────┐     ┌──────────┐    ┌──────────┐   ◀ 网关无状态副本
        │ Gateway  │     │ Gateway  │    │ Gateway  │     （水平扩展）
        │  #1      │     │  #2      │    │  #N      │
        └────┬─────┘     └────┬─────┘    └────┬─────┘
             │                │                │
             └────────────────┼────────────────┘
                              │
            ┌─────────────────┼──────────────────┐
            ▼                 ▼                  ▼
       ┌─────────┐      ┌──────────┐       ┌──────────────┐
       │ Redis   │      │PostgreSQL│       │ ClickHouse   │
       │ 指标/熔断│      │ 配置/日志│       │ (日志,可选)  │
       │ 限流/缓存│      │ 关系数据 │       │              │
       └─────────┘      └──────────┘       └──────────────┘

   Admin Web (Next.js) 可独立静态部署，经 Admin API 访问后端。
   代理出口池（可选）：每 Provider/代理一个独立出口 IP，绕过上游对来源的限制。
```

- **无状态**：任一副本都能服务任一请求；流式请求由接管的那个副本独占完成。
- **配置一致性**：所有副本订阅同一 Redis 失效广播，秒级同步配置。
- **可观测**：指标进 Redis（实时面板）、日志异步进 PG/ClickHouse（查询/审计）。

## 5. 关键架构权衡

| 权衡点 | 决策 | 代价 / 说明 |
|--------|------|-------------|
| IR 富表达 vs 简单 | 选富表达（block 模型） | Adapter 代码量略大，但换来无损转换与可扩展 |
| failover 时机 | 仅首字节前 | 牺牲部分容错，换取流的正确性 |
| 配置热更新 | 内存缓存 + 广播失效 | 有 ~1s 收敛窗口；可接受 |
| 指标存储 | Redis 滑动窗口 | 重启丢失实时窗口（可从 PG 日志重建阈值，不重建历史曲线） |
| 流式转换 | 边收边转，不缓冲整段 | 低延迟；要求 Adapter 是真正的流式状态机 |
